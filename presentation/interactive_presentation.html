1. Introduction & Background Collaborative Documentation Challenges

In modern professional, academic, and business settings, **meetings are the main driver of collaborations and decisions**.

However, capturing and acting on the information produced during these meetings remains highly inefficient.

Traditional Documentation Flaws:

  • **Inconsistent Notes:** Subjective and highly variable.
  • **Time Sink:** Reviewing transcripts/audio takes hours.
  • **Out of Sync:** Meeting actions are separate from tracking tools, leading to forgotten commitments.

2. Problem Statement The Capture-to-Action Gap

"How can we automatically convert raw, multilingual meeting media into structured, trackable action items without exposing API keys?"

🔊 Import Complexity

Handling local audio/video uploads alongside links to cloud recordings (Drive, YouTube, etc.) seamlessly in a single app.

🧠 AI Formatting

Instructing LLMs to reliably parse audio transcriptions into a strict, type-safe JSON schema instead of plain text.

🔒 Client Security

Ensuring Gemini API keys are never embedded in the client APK/IPA, protecting it from reverse engineering.

3. Project Objectives Target Solutions

  • Automated outcome parsing: Generate Summaries, chronological Minutes of Meetings, and Decisions.
  • Actionable task generation: Automatically extract tasks, assignees, and deadlines into a database.
  • Unified tracking center: Expose a global user dashboard for marking tasks complete and updating metadata.
  • Multimodal ingestion: Import raw MP3/MP4 files or paste public links (e.g. YouTube, cloud links).
  • Language preservation: Keep the source meeting language (e.g., Arabic meeting outputs Arabic documents).
  • Secure serverless routing: Proxy API requests securely through cloud middleware logic.

5. Clean Architecture Layers Logical Code Organization

Layer Responsibilities:

  • Presentation: Screen widgets and State Notifiers. Completely decoupled from database libraries.
  • Domain: Strict model contracts (Entities), data models (parsing JSON maps), and exceptions.
  • Data: Repository implementations which fetch raw data and map it to domain models.

Data Ingestion Flow:

  1. User imports raw file → **FilePickerService** reads bytes.
  2. Draft meeting created → **Cloud Firestore** triggers listener.
  3. Request proxy to **Firebase AI Logic** (serverless) → **Gemini** analyzes audio.
  4. Outputs strict JSON → Client receives parsed data structure.

7. Data Model & ERD Firestore Database Schema

Key Entities & Fields:

  • **AppUser:** Namespace identifier, profile parameters.
  • **Meeting:** Contains metadata (title, status, date) and summaries/minutes text blocks.
  • **Decision:** Extracted agreements (contains text and owner).
  • **MeetingTask:** Scoped action items (title, description, assignee, dueDate, priority, status).
**Note on Deletion:** When a meeting is deleted, all subcollections (decisions, tasks) and their corresponding global tasks are deleted in an atomic write batch.

8. Sequence Model: Authentication State Management

Authentication Life Cycle:

  1. User inputs credentials into **AuthView**.
  2. View triggers **AuthNotifier** (Riverpod controller).
  3. Notifier requests **AuthRepository** layer.
  4. Repository queries **Firebase Auth** securely.
  5. Successful credentials update notifier state, triggering immediate navigation redirect via **GoRouter**.

15. Security & Write Atomicity Robust Data Design

🛡️ Firestore Security Rules

service cloud.firestore {
  match /databases/{database}/documents {
    match /users/{userId}/{document=**} {
      allow read, write: if request.auth != null 
        && request.auth.uid == userId;
    }
  }
}

Ensures strict per-user data isolation. Users cannot view or edit meeting records belonging to others.

🔄 Atomic Batch Writes

  • **Atomic Double-write:** When Gemini generates tasks, the app writes the records under `/meetings/{meetingId}/tasks/{id}` and `/tasks/{id}` inside a single **WriteBatch** operation.
  • **Tombstone Cleanup:** When deleting a meeting, all sub-collections and user-level duplicated tasks are updated/deleted atomically to prevent orphaned data.
